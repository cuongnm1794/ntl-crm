Tổng quan kiến trúc hệ thống

Architecture

Hệ thống quản lý và gửi mã thưởng (voucher) Nestlé qua Zalo ZNS (GapOne) và SMS (Gateway 8069). Tích hợp Call Center Alohub. Stack: NestJS 11 + React 18 (Ant Design 5) + PostgreSQL + Redis + BullMQ.

    Bấm vào bất kỳ node nào để xem chi tiết. Chọn chip phía trên để xem luồng dữ liệu theo chức năng.

    Global Guards

    JwtAuthGuard → RolesGuard → NonceGuard → RateLimitGuard (100 req/60s via Redis). @Public() bỏ qua JWT. @Roles() kiểm tra quyền.

    Global Interceptors

    LoggingInterceptor (correlationId). IdempotencyInterceptor (Redis cache). StripVoucherCodeInterceptor (che mã voucher).

    4 BullMQ Queues

    voucher-import · gapone-send · export-jobs · mt-8069. 3 retries, exponential backoff 3s, Redis backend.

    Luồng đăng nhập & Xác thực

    5 endpoints

    Mục đích

    Xác thực admin bằng JWT (access + refresh token). Access token ngắn hạn, refresh token 7 ngày với secret riêng. tokenVersion để thu hồi toàn bộ token khi đổi mật khẩu hoặc disable tài khoản.

    Actor tham gia

    admin pm cs tdv brand

    Luồng chính

    POST /auth/login→ bcrypt verify→ JWT (sub, role, tv)→ Zustand persist
    401 response→ POST /auth/refresh→ verify refresh JWT + tv→ new access token

    API endpoints

    MethodPathAuthMô tả
    POST/auth/loginPublic + Throttle 5/minĐăng nhập → access + refresh token. Payload: {sub, role, tv}
    POST/auth/refreshPublicRefresh token → access mới. Kiểm tra tokenVersion
    GET/auth/meJWTLấy profile admin hiện tại
    PATCH/auth/me/passwordJWTĐổi mật khẩu → tokenVersion++ → thu hồi toàn bộ token cũ

    Rule / Validate

    • TokenVersion: Mỗi lần đổi password/disable admin → +1 → token cũ vô hiệu.
    • JwtStrategy: Kiểm tra admin tồn tại, isActive=true, tokenVersion khớp.
    • Refresh Token: 7 ngày, secret riêng (JWT_REFRESH_SECRET).
    • Frontend Axios interceptor: Tự động refresh khi 401 → retry request gốc. Refresh fail → clear auth → redirect /login.
    • Rate Limit: 5 request/phút cho login endpoint.
    • Seed accounts: Nếu RUN_SEED=true, tạo 5 tk mặc định: admin/admin123, pm/pm123456, cs/cs123456, tdv/tdv123, brand/brand123.

    04

    Luồng nhập mã Voucher (Import)

    8 endpoints

    CS upload file Excel chứa danh sách voucher → VoucherImportProcessor xử lý 5 phase: parse → validate DB → batch insert → enqueue send → error XLSX.

    5 Phase xử lý trong VoucherImportProcessor

    Phase 1 Read & Parse Excel: Đọc file bằng xlsx library. Kiểm tra 11 cột bắt buộc: Tháng, StartDate, EndDate, Khu vực, Distributor, Customer, Customer name, Contact, RedeemCode, Value, Mapping. Validate từng dòng: phone (VN mobile regex), month (1-12), voucherCode (6 ký tự), dates (DD/MM/YYYY), trùng code trong file, ZNS template field lengths.

    Phase 2 Pre-validate DB: Kiểm tra voucher code trùng trong vouchers table bằng WHERE "voucherCode" = ANY($1).

    Phase 3 Batch Insert (transaction): Chia thành batch 100. Trong transaction: insert Voucher (status=PENDING), sync customer_phone_map, tạo VoucherSendLog (status=PENDING, trigger=auto, channel=ZNS).

    Phase 4 Enqueue Send: Với mỗi voucher đã insert thành công, thêm job vào gapone-send queue.

    Phase 5 Finalize: Generate error XLSX (ExcelJS WorkbookWriter). Flush errors mỗi 500 dòng. Update job status + audit log.

    API endpoints

    MethodPathRolesMô tả
    POST/vouchers/import-jobscsUpload Excel (.xlsx/.xls/.csv, max 10MB) → tạo ImportJob (PENDING) → enqueue voucher-import
    GET/vouchers/import-jobscsList jobs + auto-poll mỗi 5s nếu còn job active
    GET/vouchers/import-jobs/:idcsJob detail: progress, stats, vouchers, error summary, send stats
    GET/vouchers/import-jobs/:id/errorscsPaginated error list (row, message, voucherCode, phone)
    GET/vouchers/import-jobs/:id/errors.xlsxcsDownload error XLSX file
    GET/vouchers/import-jobs/:id/filecsDownload file gốc đã upload
    GET/vouchers/import-jobs/templatecsDownload template Excel
    GET/vouchersadmin csList vouchers (search phone/code, filter status/date/importJobId)

    Edge cases & Validate

    • File sai định dạng: Lỗi ngay — không parse được Excel → FAILED.
    • Hàng Excel trống: Bỏ qua, không tính lỗi.
    • Voucher code trùng trong file: Ghi nhận dòng lỗi + dòng xuất hiện đầu tiên.
    • Voucher code trùng DB: Phase 2 phát hiện → mark FAILED + error XLSX.
    • Số điện thoại không hợp lệ: Regex VN_MOBILE_PHONE_REGEX — phải là số di động VN.
    • ZNS field quá dài: ten_chuong_trinh(50), so_dien_thoai(15), ten_CH(50), so_luong_qua(50), ma_tra_thuong(20), ngay_bat_dau(20), ngay_ket_thuc(20).
    • Import crash giữa chừng: Job FAILED, audit log ghi nhận, error XLSX được generate.
    • Campaign guard: Kiểm tra CAMPAIGN_START_AT / CAMPAIGN_END_AT — chặn ngoài chiến dịch.

    Luồng gửi Voucher qua GapOne (G1)

    5 endpoints

    Worker GaponeSendProcessor nhận job từ queue gapone-send → build ZNS template → OAuth2 → POST GapOne a2p gateway. Có SMS fallback, mock mode, whitelist.

    Xử lý trong GaponeSendProcessor

    1. Set voucher → SENDING
    2. Build ZNS template data (7 params)
    3. OAuth2 client_credentials → GapOne (cache 60s)
    4. POST {gatewayUrl}/a2p-request
    5. Nếu accepted (có messageId) → tạo VoucherSendLog (PENDING)
    6. Nếu mock mode → success ngay
    7. Nếu failed → voucher → FAILED

    Mock & Whitelist & SMS Fallback

    • Mock mode: GAPONE_MOCK=true → giả lập thành công, không gọi API thật.
    • Whitelist: Ngoài production, chỉ gửi đến số trong GAPONE_SEND_PHONE_WHITELIST.
    • SMS fallback: Nếu GAPONE_SMS_BRANDNAME được cấu hình → gửi SMS (strip dấu tiếng Việt).

    Luồng nhận Delivery Status (Webhook + Cron)

    Webhook path

    GapOne push DN→ POST /gapone/webhook/delivery→ resolveFromDeliveryStatus()→ Update SendLog + Voucher

    Cron 60s→ Query send_logs pending >10min→ Batch 50 messageIds → queryDeliveryBatch()→ applyDelivery()

    Fallback khi webhook không đến. Đảm bảo không bỏ sót delivery status.

    Luồng tra cứu CSKH

    3 endpoints

    CS/TDV nhập SĐT → CskhService.lookup() resolve customer codes → build tabs + timeline từ 4 nguồn dữ liệu (voucher, g1_send, call, case).

    Chi tiết GET /cskh/customers?phone=...

    Step 1 Normalize phone: Chuyển +84/84 → 0 prefix.
    Step 2 Resolve customer context: Tra customer_phone_map → lấy customer codes. Tra phone_change_logs JOIN vouchers → bổ sung customer codes. Tìm vouchers trực tiếp theo phone → bổ sung và tách legacy vouchers (không có customer code).
    Step 3 Build tabs: Mỗi customer code → 1 tab: summary (total + byStatus), event counts (voucher, g1_send, call, case trong 90 ngày), phone change history, danh sách voucher.
    Step 4 Legacy tab: Voucher không có customer code → tab "Không có Customer".

    Chi tiết GET /cskh/customers/timeline?...

    Step 1 Resolve timeline context: Chọn 1 customer (hoặc legacy). Liên kết vouchers + phones.
    Step 2 Fetch 4 source batches: voucher (từ voucher list), g1_send (từ voucher_send_logs), call (từ alohub_calls theo phones), case (từ alohub_cases theo phones).
    Step 3 Merge & Paginate: Sort desc theo createdAt. Cursor-based pagination (base64 của {createdAt}|{type}|{id}). Max 100 events/request.

    API endpoints

    MethodPathRolesMô tả
    GET/cskh/customerscs tdvTra cứu theo phone → tabs per customer code
    GET/cskh/customers/timelinecs tdvTimeline 4 nguồn + cursor pagination (max 100 events)
    PATCH/cskh/customers/phoneadminĐổi SĐT liên hệ (idempotent, distributed lock Redis)

    Luồng MO/MT (SMS 8069)

    2 endpoints

    Khách hàng nhắn "Nestle T5" → 8069 → Hệ thống trả lời voucher

    Step 1 Gateway 8069 forward MO SMS tới POST /momt/resend-voucher (public).
    Step 2 Save MO log vào mo_logs.
    Step 3 Parse message: Regex /^nestle\s+(0?[1-9]|1[0-2])$/i → lấy tháng.
    Step 4 Check campaign time: So sánh VALID_DATE_START / VALID_DATE_END.
    Step 5 Find voucher: Tìm theo phone + month (ưu tiên send_log.status = SUCCESS).
    Step 6 Enqueue MT: Thêm job vào mt-8069 queue.
    Step 7 ResendVoucherProcessor: POST GW_URL_8069 (Basic auth) → lưu MT log vào mt_logs.

    API

    MethodPathAuthMô tả
    POST/momt/resend-voucherPublicNhận MO SMS từ gateway 8069
    GET/momtadmin tdvTra cứu MO/MT history (LEFT JOIN)

    Các điểm rủi ro & Cần làm rõ

    1. Module Tickets bị disable

    Controller Tickets đã code đầy đủ nhưng bị comment khỏi module import. Service vẫn tồn tại. Cần xác nhận lý do và kế hoạch kích hoạt lại.

    2. Xác thực webhook GapOne + Alohub chưa rõ cơ chế chi tiết

    Cả hai webhook đều dùng public endpoint với key nhưng cơ chế xác thực cụ thể (header/query param/thuật toán) chưa được document rõ trong code.

    3. Nội dung SMS template (MT) chưa rõ

    Các template MT_INVALID_SYNTAX, MT_DATE_START, v.v. được cấu hình qua biến môi trường nhưng nội dung cụ thể chưa xác định.

    4. Campaign ended_early bị hardcode

    ended_early trong StatusService không tự động set khi voucher hết. Cần xác nhận logic này có được sử dụng không.

    5. GapOne mock mode & whitelist

    Ngoài production, chỉ gửi đến số trong whitelist. Cần xác nhận danh sách whitelist và cơ chế cập nhật.

    6. Không có alerting/notification

    Hệ thống chưa có cơ chế thông báo khi import lỗi, send thất bại hàng loạt, hay các sự kiện quan trọng. Chỉ có audit log thụ động.

    7. Concurrent phone change — Distributed lock

    Redis lock được dùng cho changeCustomerContactPhone. Cần xác nhận timeout lock và xử lý khi lock fail.

    8. Cursor pagination timeline

    Cursor format {createdAt}|{type}|{id}. Cần kiểm tra độ chính xác khi nhiều event cùng timestamp.

    9. ReportsHub frontend bị tạm ẩn

    Trang /admin/reports bị comment trong router frontend với ghi chú "temporarily hidden".

    10. XSS — dữ liệu từ Excel/SMS

    Cần xác nhận dữ liệu từ Excel import và SMS được sanitize đúng cách trước khi lưu/hiển thị.
